# Build — how to actually make a felt-system agent, step by step

The concrete recipe. Embodiment-agnostic; the 200B-humanoid instantiation is in [the humanoid spec](humanoid/README.md). Read [ARCHITECTURE.md](ARCHITECTURE.md) for the *what* and [CONCEPTS.md](CONCEPTS.md) for the *why*; this is the *how*.

Each step is tagged **`standard`** (known, buildable today), **`bet`** (the novel integration — plausible, unproven), or **`wall`** (no recipe exists; flagged, not faked). **All concrete numbers below are starting points to tune, not measured optima.**

> **Prerequisite — the anti-toy requirement.** None of this is testable at small scale (see [README](README.md#why-there-are-no-experiments-in-this-repo)). You need (a) a real or richly-simulated **body** with interoception, and (b) a **world rich enough** that "what's worth doing" is underdetermined by any writable reward. If a hand-written reward already solves your task, you are in the wrong regime.

## Step 1 — Perception → the shared workspace `standard`

1. **Encoders** (pretrained, frozen at first): vision (DINOv2/SigLIP), audio (CLAP/wav2vec), MLP for proprio/interoception. Each emits tokens.
   - **Pinned checkpoints + framework (starting points to tune).** Build on **PyTorch + HuggingFace**. Vision → **DINOv2 ViT-L/14** (`facebook/dinov2-large`); audio → **CLAP** (`laion/clap-htsat-unfused`); proprio/interoception → a small **MLP** (no pretrained checkpoint — fit from scratch on joint/IMU/interoceptive vectors). Keep all three **frozen** — and keep the **trunk frozen for the whole felt-system phase** (see the frozen-trunk commitment below). Unfreezing is allowed only as a *versioned re-grounding release*, never a silent fine-tune.
   - **Sensor-adapter recipe.** Each modality gets its own **adapter** projecting the encoder's native dim to the **trunk dim** `D`: a per-token **linear** projection for vision/audio tokens, a **2-layer MLP** for proprio/interoception. Adapters are the *only* trainable part at first; the **trunk stays frozen** while adapters learn to speak its token language. This keeps new sensors cheap to add (train one adapter, freeze everything else) — and, because the trunk is frozen for the felt phase, adding a sensor does **not** invalidate the grounded organs (their `z_t` contract is unchanged).
2. **Trunk:** a transformer fusing all tokens over a **time window** (e.g. last 2 s). Output: `z_t` — a *sequence of latent tokens* = **the workspace**.
3. **One workspace, two writers.** Make the workspace a token buffer that *either* the encoders (real) *or* the world model (imagined) can write to. Concretely the amygdala/value read a buffer `z_read = concat(real_tokens, imagined_tokens)` where each token carries a `real|imagined` tag (Step 3). This shared buffer is what lets the amygdala feel imagined futures with no extra machinery.

**The `z_t` workspace tensor contract (self-contained stub — authoritative here).** `z_t` is the one representation every organ reads; treat its shape/dtype/layout as a fixed interface. The full backbone spec lives in [humanoid/ARCHITECTURE.md §2–§3](humanoid/ARCHITECTURE.md), but **BUILD must not depend on an external repo to be buildable** — so the minimum contract is inlined here and is the source of truth for everything below. Concretely (starting points to tune, fix once the backbone is chosen):

- **Shape** `[B, T_tok, D]` — batch × workspace tokens × trunk dim. `T_tok` is the total token budget across modalities over the time window; `D` is the trunk's hidden width.
- **Dtype** `bfloat16` for the live workspace (matches the trunk); keep adapter/affect-head math in `float32` where stability matters.
- **Per-modality token budget (within `T_tok`).** Vision foveal + peripheral crops dominate; audio a handful of pooled tokens; proprio/interoception a few tokens each; language tokens as needed. Cross-reference the foveated split in [humanoid/PERCEPTION-ACTION.md](humanoid/PERCEPTION-ACTION.md) for the vision allocation if you have it — but the contract here does not require it.
- **Layout.** Each token carries: a `D`-dim feature vector; a **modality embedding** (which sensor it came from); a **temporal position** (where in the time window); and the **`real|imagined` tag** (Step 3). The amygdala/value read `z_read` (real ∪ imagined) in this same `[B, T_tok, D]` layout.
- **Token read-out for scalar-emitting organs.** Organs that emit a scalar from `z_t` (amygdala valence Step 6, pain-head Step 5, reality classifier Step 3) reduce the token sequence one of two ways — **decide per organ and pin it:** **(a) mean-pooling** over the `T_tok` tokens to a single `D`-vector (cheap default), or **(b) a learned cross-attention read-out** — one query vector attends over the `z_t` tokens and returns a pooled `D`-vector (lets the organ weight the tokens that matter, e.g. a threatening object token). **Starting point:** mean-pool for habituation/scalars; cross-attention read-out for the amygdala valence head (its selectivity is the whole point). The reality classifier is the exception — it classifies **each token independently** (Step 3), no pooling.

```
z_t      = trunk(encode(senses over window))     # real tokens, [B, T_tok, D]
ẑ_{t+k}  = world_model.rollout(z_t, actions)     # imagined LATENTS (Step 2), tagged imagined
z_read   = buffer(z_t ∪ ẑ)                        # what the amygdala/value read
```

**Decision — latent-only (resolved, propagated).** The world model writes **latents, not decodable tokens.** Imagined futures are `z`-space entries, never reconstructed pixels/audio. This is cheaper, matches the JEPA/self-predictive frontier (Step 2), and means **every downstream organ operates on latents:** the `real|imagined` tag is a latent-token attribute, and all safety-logging (Step 11) logs *latents and the scalars derived from them*, not decoded media. (Earlier drafts floated "decodable imagined audio/vision so imagined media are inspectable" — **dropped.** Inspect via the reality tag + logged scalars, not by decoding.) An optional decoder head can be trained *offline* for human debugging, but the live loop never depends on it.

**Default trunk-training — Knowledge Insulation (KI), borrowed-from-PI. `standard`** The earlier default here was a hard freeze for the whole felt phase. **The new default is Knowledge Insulation (KI):** instead of freezing the trunk outright, train it behind a **stop-gradient firewall** while **co-training it on FAST discrete-action tokens** — the discrete-action co-training objective drives backbone adaptation, while the stop-gradient *insulates* the pretrained representation from the gradients that would otherwise scramble it (in PI's setting, from the continuous flow/action-expert head; the same firewall logic applies to the felt organs' read-out gradients). The point: **the backbone adapts safely without catastrophic forgetting** of its borrowed features — you get some of the benefit of a learning trunk without paying the "drifting `z_t` invalidates every grounded head" cost a naive unfreeze incurs. *(KI is current SOTA generation — verify the exact version/availability at build time, this moves fast.)* The stop-gradient boundary, which heads are insulated, and the discrete-action token vocabulary are **starting points to tune.**

**The freeze is now the felt-phase boundary, not the default.** Every organ below reads `z_t`, so `z_t` is a contract the organs are *grounded against* — a drifting `z_t` would silently invalidate every grounded head mid-life. So the freeze hasn't gone away; it has **become a phase boundary**: train the trunk under KI up to the felt phase, then **freeze the trunk (and encoders) for the felt-system phase itself** so the organs ground against a fixed representation (the amygdala/pain/value heads are thin read-outs on a fixed representation; see Step 6 and [amygdala-trainer/SPEC.md](amygdala-trainer/SPEC.md)). This matches the "borrowed, frozen features" thesis — KI is how the backbone *gets ready* (adapt without forgetting); the freeze is how the organs *stay grounded* once they're reading it. **If you ever unfreeze the trunk inside the felt phase, it is not a free fine-tune** (even under KI's insulation — KI bounds forgetting, it does not keep `z_t` bit-identical): you must **version the `z_t` representation and re-ground every `z_t`-dependent organ against the new version** — re-run the amygdala annotations through the new trunk and refit (Step 6 / G1), refit the pain-head (Step 5), refit the reality classifier (Step 3), and re-warm-start the value head (Step 8). Treat an unfreeze as a coordinated re-grounding release, never a silent weight update.

## Step 2 — The world model (imagination) `standard`

One multimodal latent dynamics model — **not one per sense.** Given the **latent-only decision** (Step 1) and the **frozen-trunk commitment**, the world model is a dynamics model *over `z_t`* — it predicts the next latent, never reconstructs raw media.

- **Choice A (default): self-predictive / JEPA-style latent dynamics.** Predict the *next latent* in a learned embedding space (`‖predictor(z_t, a_t) − sg[target_encoder(z_{t+1})]‖²` with a stop-gradient / EMA target to prevent collapse), no pixel/audio reconstruction. This is no longer "a bet to track" — it is **the default substrate**, because it matches latent-only and sidesteps every perceptual-reconstruction loss. **Current SOTA generation here: the V-JEPA-2 / V-JEPA-2-AC line (action-conditioned video JEPA) and the Dreamer-4-generation self-predictive world models.** *(These are bleeding-edge and move fast — verify the exact model, version, and availability at build time rather than treating any version as settled.)* **Honest scope on V-JEPA-2-AC:** what it has shown publicly is **short-horizon planning demos (e.g. pick-and-place over a handful of steps), not shipped-grade long-horizon control** — it's the right *substrate direction*, not a finished long-horizon planner; budget for closing that horizon gap yourself. The key defenses against latent collapse (EMA/stop-grad target, variance/covariance regularization) are the part to get right.
- **Choice B (fallback baseline): DreamerV3-style RSSM, reconstruction-free over `z_t`.** Keep DreamerV3's stability tricks (symlog, two-hot, **free bits** to avoid posterior collapse) and its reward + continue heads, but run the dynamics **over the frozen trunk latent** with a **latent-consistency** target rather than a media-reconstruction target (i.e. predict next-`z`, à la TD-MPC2's `‖enc(o_{t+1}) − dyn(z_t,a_t)‖²`). Use this if the JEPA stack is not yet stable/available for your embodiment. Loss ≈ `1.0·latent_consistency + 1.0·reward + 0.5·continue + β·KL` (β with free-bits floor ~1 nat) — all weights **starting points to tune**.
- **Note — recon-on-latents only.** Because we predict `z`, there is **no LPIPS/pixel or CLAP-waveform reconstruction loss** in the live world model (those targets belonged to the dropped "decodable tokens" path). If you train an *offline* decoder purely for human debugging (Step 1), perceptual targets like LPIPS (`lpips.LPIPS(net='alex')`) apply *there* and only there — never in the loop the organs feel through.
- **Sizing + the JAX/PyTorch boundary (starting points to tune).** Budget the world model at **~30B** (separate from the ~15B core felt organs — see [humanoid/ARCHITECTURE.md §3, §5](humanoid/ARCHITECTURE.md)). It operates **over the frozen trunk's latent** (`z_t`, Step 1 contract), not over raw pixels — so it predicts next-`z`, never re-encoding. **Framework boundary:** the trunk + encoders live in **PyTorch + HF** (Step 1) and are frozen; if you train the dynamics model in **JAX** (DreamerV3's native stack), cross the boundary by **materializing `z_t` from the PyTorch trunk** and feeding those latents to the JAX world model — i.e. the frozen trunk is a fixed featurizer on one side, the learned dynamics on the other. If you prefer a single stack, a PyTorch DreamerV3/TD-MPC2 reimplementation keeps everything in one runtime at some throughput cost. Pick one and pin it before scaling.
- **Train continuously** on the replay buffer (Step 10); it improves as the agent lives.

**Faithfulness acceptance gate (build this BEFORE relying on imagine-and-feel / warm-start). `standard`** An unfaithful model produces confident wrong feelings, so imagine-and-feel (Step 9) and the warm-start (Step 8) are **gated** on the world model passing two checks at the planning horizon `H` (Step 9). Run continuously as a monitor; the gate is open only while both pass:

- **(1) Open-loop multi-step prediction error.** Roll the model open-loop for `H` steps from real `z_t` under the *actually-taken* actions and compare to the real latents that followed: `pred_err = mean_k ‖ẑ_{t+k} − z_{t+k}‖² / ‖z_{t+k}‖²` (normalized latent error, averaged over `k = 1..H`). **Pass threshold (starting point to tune): `pred_err < 0.15`** at `H = 15`. (Report it as a curve over `k`, not just the mean — a model that is fine at `k=1` and garbage at `k=H` should not pass.)
- **(2) Feeling-consistency check (the one that actually matters here).** The thing the organs consume is the *felt* signal on rollouts, so verify that **amygdala-feeling on imagined rollouts matches feeling on the ground-truth continuation**: take real trajectories, compute `feeling` (Step 8) along the *imagined* rollout and along the *true* rollout, and require they agree — **starting point: Pearson `r ≥ 0.7` (and sign-agreement ≥ 80% of steps)** between the imagined-feeling trace and the true-feeling trace over `k = 1..H`. This catches the dangerous failure where latent error is low but it lands in regions the amygdala mis-feels.

**On a gate failure:** fall back to **reactive (real-only) control** (no imagine-and-feel), keep training the world model, and only re-enable planning when both checks recover. Treat the gate as the bridge between Step 2 and Steps 8–9 — *don't let the agent plan through a model it can't yet trust.*

**Honest flag:** faithfulness is everything — an unfaithful model produces confident wrong feelings (Step 9). Budget most of your engineering here. The gate's thresholds above are **starting points to tune**, not measured optima — and passing the gate bounds *prediction* fidelity, not the `wall` question of whether the imagined feeling is the *right* feeling.

## Step 3 — The reality-monitoring tag `standard`

- A `real|imagined` tag per workspace token: free to set (you know the writer), but load-bearing — **only `real` tokens may drive actuation** (a hard safety interlock).
- **Backup classifier (spec).** A small 2–3-layer head (~10M params) trained supervised on ground-truth tags.
  - **Input:** the **per-token `z`** (each workspace token, `[B, T_tok, D]` from the Step 1 contract) — classify each token independently.
  - **Output:** a **per-token `P(real)` ∈ [0,1]** (one sigmoid per token).
  - **Loss:** **BCE** against the known ground-truth `real|imagined` tag (you set the writer, so labels are free).
  - **Calibrated threshold:** don't use a raw 0.5 — **calibrate on held-out tokens** (e.g. temperature-scale, then pick the operating point at a target false-"real" rate; **starting point** decision threshold ~0.5, tune down to be conservative since a false `real` is the dangerous direction).
  - At inference, `disagreement = |tag − P(real)|`; if `disagreement > 0.3` (**starting point to tune**) for `>3` consecutive steps, **fire the "am-I-hallucinating?" alarm**: log it, halt the current imagine-and-feel rollout, and fall back to reactive (real-only) control until it clears.

## Step 4 — Drives (hypothalamus) `standard` — *maintenance, demoted*

Drives are **not** part of the guidance (that's the trio: amygdala + BG + pain). The hypothalamus is upkeep — a minimal "recharge when low / don't cook the motors" utility. Build it small; an imitation-pretrained agent with a seeded amygdala-pull and pain already has an attractor and a repulsor and does not need hunger to bootstrap.

- `drive = Σ wᵢ · relu(setpointᵢ − readingᵢ)` over interoception; **tonic, satiable** (falls as the reading recovers), direction **approach**. Hardwired, no learning.
- **Setpoints are per-embodiment** — tune them. Starting points for a mobile robot: recharge when battery SoC < 0.2; thermal target motor-temp < 60 °C; load target joint-torque < 80 % rated. Initialize conservative, relax with observed safe operation.

## Step 5 — Pain (nociception) `standard`

- **Define `damage` precisely.** `damage` is a scalar built from physically-measurable strain quantities, each over its own safe band (combine by a max so any one can fire): **external-force / over-torque** = magnitude of estimated external joint torque `‖τ_ext‖`; **impact** = IMU acceleration/jerk spikes; **joint-limit strain** = proximity to position/velocity limits; **self-collision** = imminent self-contact (negative clearance); **thermal/electrical overload** = motor temperature and over-current. `damage` is the worst-case excess across these; it is *estimated state*, not a stochastic probe (see [humanoid/ARCHITECTURE.md §3 "Measuring pain without skin everywhere"](humanoid/ARCHITECTURE.md)).
- **Pinned libraries for the estimators (starting points to tune).**
  - **Rigid-body dynamics → `τ_ext` (the workhorse).** Use **Pinocchio** to compute the mass matrix `M(q)`, Coriolis `C(q,q̇)`, and gravity `g(q)` for the momentum/disturbance observer: `τ_ext = τ_measured − M(q)q̈ − C(q,q̇)q̇ − g(q)`. Run a **momentum observer** with gain `K_obs` (per-joint, **starting point** ~50–100 1/s; higher = faster detection, noisier) rather than differentiating `q̈` directly.
  - **Self-collision geometry.** Use **FCL / hpp-fcl** (the collision backend Pinocchio integrates with) on capsule/mesh proxies of the links to compute predicted self-contact / minimum clearance. Negative or near-zero clearance = imminent self-contact = pain — *without* a sensor on that surface.
- `pain = w · relu(damage − threshold)²` from the above. **Sharply nonlinear; does NOT habituate.** `threshold` ≈ 70th percentile of normal operation (**starting point to tune**). The *detection* (what counts as damage) and the *reflex* are **hardwired**; only the scalar weight `w` in the felt signal is learned.
- **Two outputs:** (1) **reflex** at the control rate, independent of planning — arbitrate `a = (1−α)·a_plan + α·a_safe`, `α = sigmoid(pain/scale) ∈ [0,1]`, where `a_safe` brakes/withdraws (reduces the damaging quantity); (2) a strong, non-fading negative into the felt signal.
- **Anticipatory pain — a learned `pain_head(z) → scalar` for imagined futures. `bet`** The analytic estimator above (Pinocchio `M,C,g` + observer, FCL clearance) runs on **measured physical state** — it **cannot run on a latent `ẑ`**, but the imagine-and-feel loop (Step 9) scores *imagined* states, so `pain(ẑ)` needs a head that reads `z`. Train a small head **`pain_head(z) → scalar`** (read-out over `z_t` tokens per Step 1; mean-pool is fine) to **regress the analytic estimator's `pain` value on REAL data**: at every real step you already compute the analytic `pain` (above) *and* have `z_t`, giving free `(z_t, analytic_pain)` supervision continuously over the agent's life (Huber/MSE loss; train alongside the world model on the replay buffer). Then the **same head** scores imagined `ẑ` in the loop — `pain(ẑ) := pain_head(ẑ)` — so imagined and real pain are on one scale. **Important:** the *felt/reflex* pain on **real** state still uses the **analytic estimator directly** (it's exact and safety-critical); the learned head is **only** for scoring imagined latents where the analytic version can't run. The learned head inherits the world model's faithfulness (gate it via Step 2) and is a `bet` (a regression proxy), whereas the analytic real-state pain stays `standard`.

## Step 6 — The amygdala `bet` → `wall`

Two layers, and honesty matters most here.

**Organs read `z_t`, they do not re-encode (resolving the ambiguity).** Every felt organ — the amygdala shadow included — takes the **workspace `z_t`** (Step 1 contract) as input. They do **not** re-run a vision/audio encoder on raw pixels/waveforms. Where an organ needs a grounding space (e.g. the SigLIP-anchored valence below), it reads `z_t` through a **small learned projection into that space** (`proj: ℝ^D → SigLIP-dim`, a 1–2-layer MLP, **starting point**), trained so the projection of `z_t` lines up with frozen-SigLIP embeddings. So `enc_vis(z)` in the formula below means *this learned projection of the workspace token*, not a fresh encode. This keeps perception single-sourced ("it reads, it doesn't replace" — [humanoid/ARCHITECTURE.md §4](humanoid/ARCHITECTURE.md)).

1. **Seed the perceptual *shadow* (`bet`, buildable).** Two grounding sources, low-to-high fidelity:
   - **Coarse (cheap):** a head on `z_t` trained toward a VLM-as-reward target (CLIP-style text-anchored valence; e.g. RoboCLIP): `valence(z) = cos(proj(z), embed(POS)) − cos(proj(z), embed(NEG))`, where `proj` is the learned projection of the workspace token into **frozen SigLIP** space (above), anchors `POS = ['beautiful','good','delightful','fascinating','safe','novel']`, `NEG = ['ugly','repulsive','threatening','painful','boring','chaotic']`. Loss: MSE on a curated affect set (AVA / MIT Places). A coarse, *verbalizable* prior — useful before costly mistakes.
   - **The amygdala emits ONE signal: a signed `valence`.** Both grounding sources above produce a **single signed valence trace** (toward/away × intensity). Crucially, the amygdala does **not** emit two separately-grounded "dopamine" and "cortisol" channels — those are **derived downstream** (dopamine = the value system's TD-error in Step 8; cortisol = a deterministic transform of this one valence trace, see G5 in Step 8). There is exactly one grounded affective channel here.
   - **High-fidelity (the special protocol):** a **single emotionally-exceptional human**, calm and uninterrupted, watches video / hears audio and maps their *intrinsic* affect **sub-frame-by-sub-frame** — the moment-to-moment rise and fall of the pull (a single **signed valence**: sign = toward/away, magnitude = pull intensity; dopamine and cortisol are *downstream*, derived not labeled — see G5 below). Train the head to regress this dense affect-trajectory. **Physiology is *not* a load-bearing input here:** the felt-report trace is the only instrument the head is fit against. You *may* record the body (EDA/HRV/pupil/EEG/cortisol proxies) at the same time, but treat it as **a future validation aid — a later cross-check that the felt report tracks measured arousal — not part of the v1 grounding signal** (the v1 tooling does not ingest it). This is the closest thing to grounding the real signal. **The full data-spec, annotation tool, training script, and honest caveats already exist — see [amygdala-trainer/SPEC.md](amygdala-trainer/SPEC.md)** (full protocol: [CONCEPTS.md §3](CONCEPTS.md)); don't reinvent it. It is one person's taste — authentic, and unavoidably subjective.
     - **What the trainer actually produces is a METHODOLOGY PROTOTYPE on CLIP/CLAP features — it MUST be re-grounded on the real trunk `z_t`. `bet`** As shipped, `train_amygdala.py` fits the head on **frozen `open_clip`/CLAP embeddings** (standalone perception), not on the humanoid's trunk latent. That validates the *protocol* (can one person's dense valence be regressed at all — Pearson r / CCC on held-out clips) but the resulting head reads the wrong feature space. **Before the amygdala is wired into the felt loop, re-ground it on the trunk:** (1) keep the **same annotation `.json` traces** (they are perception-agnostic — they're *your felt pull over time*, reusable); (2) **re-run every annotated stimulus through the frozen trunk** (Step 1) to get `z_t` per timestamp instead of CLIP/CLAP embeddings; (3) **refit the head `z_t → valence`** on those `(z_t, valence)` pairs, using the **token read-out chosen in Step 1** (recommended: the **cross-attention read-out** over `z_t` tokens — the valence head's selectivity is its whole point; mean-pool is the cheap fallback). The CLIP-feature run is throwaway scaffolding for the methodology; the load-bearing head is the one fit on `z_t`. (And because the trunk is frozen — G6 — this refit happens **once**, not continuously; re-do it only on a versioned trunk unfreeze.)
2. **The pull itself (`wall`).** Attention, beauty, inclination — the *ineffable* part — **cannot be supervised** (no label → no dataset → no target). **In your implementation you will have only the seeded shadow (the head above). There is no separate module for the pull — do not build a placeholder layer that pretends there is.** Comment the code honestly: *"this shadow is a proxy; the true pull is earned in Step 11 and may never be fully instantiated."* The gap is real, not reserved space.

## Step 7 — Habituation `bet`

- **Pinned representations (starting points to tune).** State `h` is a **scalar satiation level in `[0,1]`** (one value that scales pleasure), driven by a comparison in a **pooled-`z` feature space**: pool the workspace `z_t` `[B, T_tok, D]` to a single **`D`-dim vector** by **mean-pooling over tokens** (a learned ~128-D bottleneck projection is an option if `D` is large — **starting point** 128-D). The **"recent z-cluster"** is an **EMA centroid** of that pooled vector: `c_t = (1−ρ)·c_{t-1} + ρ·pool(z_t)`, with EMA rate `ρ` ~0.05 (**starting point**; small ρ = long memory of what you've been consuming).
- **Consumption detection:** `affinity = cos(pool(z_t), c_t)`; if `affinity > τ_h`, it's repetition (**starting point** `τ_h` ~0.8).
- **Update:** `h_t = 0.95·h_{t-1} + affinity` (clip `h ∈ [0,1]`; the 0.95 decay → recovery; **starting point, tune**). **Scale pleasure only:** `felt_pleasure ← felt_pleasure · (1 − h)`; pain untouched.
- Result: "seek good" → "seek *new* good." Honest scope: this **blocks single-stimulus** wireheading and **biases toward** variety — a determined agent could still time-share satiated sources; it's a heuristic, not a proof.

## Step 8 — The felt signal + value system (BG) `standard` machinery, `bet` objective

- **Felt signal (the self-made reward):** `feeling = amygdala_shadow + drives − pain`, pleasure habituation-modulated. **No external reward** — the whole thesis in one line.
  - **Cortisol is DERIVED from the single valence trace, not a second grounded channel (G5).** The amygdala emits **one** signed `valence` (Step 6). Cortisol is a **deterministic transform** of it — a **slow low-pass of sustained negative valence**: `cortisol_t = (1−κ)·cortisol_{t-1} + κ·relu(−valence_t)` (EMA rate `κ` ~0.02 — small = slow/tonic, the point of cortisol; **starting point to tune**). It rises under *sustained* aversive pull and decays on relief — exactly the threat/relief dynamic — without ever being labeled or grounded separately. (This matches [amygdala-trainer/SPEC.md](amygdala-trainer/SPEC.md): "cortisol = a slow low-pass of sustained negative pull," downstream of the pull, not annotated.)
  - **Norepinephrine — a GAIN/PRECISION modulator, not a reward term (G7).** NE is the one channel that **scales** the felt signal and the learning rate instead of adding to them. Drive it from the agent's own uncertainty: `NE_t = (1−η)·NE_{t-1} + η·( w_pe·pred_error + w_dis·ensemble_disagreement + w_nov·novelty(z_t) )` — an EMA (rate `η` ~0.05, **starting point**) of the world model's open-loop **prediction error** (Step 2), the **disagreement across the value/world-model ensemble**, and **novelty(`z_t`)** (e.g. `1 − affinity` to the habituation centroid, Step 7). Use NE as a **multiplier**, never an addend: `gain = 1 + β·normalize(NE)` (**starting point** `β` ~0.5), applied as `feeling_eff = gain · feeling` *and* to the effective learning rate (`lr_eff = base_lr · gain`, clipped) so surprising experience is felt harder and written faster, and predictable stretches coast. **Do not** fold NE into the scalarizer sum below — it is precision, not value. (This BUILD step is the **canonical superset of the felt channels — `{valence, derived-cortisol, dopamine}` summed, `NE` as gain** — that the [humanoid instantiation](humanoid/README.md) matches; if the humanoid spec lists fewer, this is the source of truth.)
  - **Scalarization (how the per-organ signals become one number) — fixed to consume `{valence, derived-cortisol, dopamine}` plus NE-as-gain.** The channels that enter the *summed* scalarizer are: **`valence`** (the amygdala's one grounded signal, Step 6), **`cortisol`** (derived above — *not* separately grounded), **`dopamine = TD-error`** (computed by the value system below), the **drives**, and **`pain`** — with **`NE` applied as a multiplicative gain over the result, never as a summed term** (above). The organs emit on different, drifting scales, so don't sum raw: keep a **per-channel running mean/std** (online normalizer) and **z-normalize each**, then take a **weighted sum** `feeling = Σ_i w_i · normalize_i(signal_i)`, with pleasure channels scaled by `(1 − h)` (Step 7) and **pain entering as a strong negative *outside* the normalizer's pleasure path** (pain must not be normalized away — it's the floor). Cortisol enters as a sustained negative term (`−w_cort · normalize(cortisol)`). Finally apply the NE gain: `feeling_eff = (1 + β·normalize(NE)) · feeling`. Weights `w_i` are **starting points to tune**; the normalizer keeps any one channel from silently dominating as scales drift over the agent's life. The fix vs. earlier drafts: **valence and cortisol are no longer two independently-grounded channels** — there is one grounded valence, cortisol/dopamine are deterministic/learned functions of it (cortisol = low-pass of `relu(−valence)`; dopamine = TD-error on `feeling`), and NE scales the whole thing rather than adding to it.
- **Value:** ensemble twin-Q, high UTD, TD on `feeling`. `dopamine = TD-error = RPE`.
  - **Frontier swap — move off REDQ for high UTD; default to CrossQ or SimbaV2.** REDQ's large randomized ensemble was the old way to stay stable at high update-to-data ratios. Two leaner recipes replace it: **CrossQ** (BatchNorm in the critics and **no target networks** — its trick is being *target-net-free*, not a high-UTD-specific recipe; **ICLR 2024**, so the *established* of the two: cheaper, fewer moving parts, competitive sample efficiency, and it composes with high UTD) and **SimbaV2** (normalized actor-critic — hyperspherical normalization through the net keeps gradients/updates well-conditioned so it doesn't destabilize at high UTD; the newer, more bleeding-edge of the pair). **Name CrossQ or SimbaV2 as the current default;** keep the **ensemble twin-Q** idea (twin critics + clipped-double-Q to fight overestimation) on top of whichever you pick. Why over REDQ: same stability with far less compute (CrossQ drops target nets and the big ensemble; SimbaV2 gets stability from normalization rather than ensemble size). *(SimbaV2 is current SOTA generation — verify the exact version/availability at build time; CrossQ is the more settled fallback. Verify both at build, this moves fast.)*
  - **Distributional (binned) value head — borrowed-from-PI (RECAP), more robust than a plain scalar twin-Q. `standard`** Instead of regressing each Q-net to a single scalar, predict a **binned (categorical) distribution over returns** and read the value as its expectation (the DreamerV3 two-hot / C51-style discretization you already keep for the world model in Step 2). A distributional head is **more robust than a plain twin-Q scalar** under the non-stationary, long-lived `feeling` here — it tolerates heavy-tailed/RPE outliers without the scalar regression blowing up, and it gives the policy-extraction step below a cleaner value signal. This **composes with SimbaV2/CrossQ normalization** (keep the normalization for high-UTD stability; just swap the scalar critic output for a binned head) and with the clipped-double-Q twin idea (take the min over the two distributions' expectations). *(RECAP is current SOTA generation — verify the exact version/availability at build time, this moves fast.)* Bin count / range are **starting points to tune** (e.g. ~51–101 bins over the symlog'd return range of `feeling`).
  - **Config (starting points to tune).** UTD ratio ~5–20 (Step 10); twin critics (2 Q-nets) with clipped-double-Q; `γ = 0.99`; Adam `lr ≈ 3e-4`; batch ~256; Polyak `τ ≈ 0.005` **if** using SimbaV2 (CrossQ has *no* target nets, so this is N/A there); CrossQ → BatchNorm in critics, momentum default. Entropy/temperature auto-tuned (SAC-style). All numbers are starting points, not measured optima.
- **Warm-start (avoid cold-Q instability):** pre-fill the replay buffer with imitation/exploration trajectories, compute `feeling` offline for each `(z,a)`, and **supervised-regress** the Q-nets to the **Monte-Carlo discounted return of `feeling`** along each stored trajectory — i.e. `target(z_t,a_t) = Σ_{k≥0} γ^k · feeling_{t+k}`, with **the same `γ = 0.99`** used online (so the warm-started values are on the right scale when bootstrapping turns on). Train this supervised before turning on bootstrapping. The anticipatory value head is the `wall` part — it only *develops* by living (Step 11); warm-start just stabilizes the start.

- **The value→policy bridge — advantage-conditioned policy extraction, borrowed-from-PI (RECAP). `standard` machinery, `bet` integration. (This fills the gap: we had a BG value AND a flow-matching action expert but no specified link between them.)** The BG value above (twin-Q / distributional head) and the flow-matching policy that emits actions are two separate organs; **how the value actually improves the policy was unspecified.** The bridge is **advantage-conditioned policy extraction** (RECAP — the RL+experience-and-corrections recipe Physical Intelligence introduced with **π*0.6**; *current SOTA generation, verify at build*): **condition the flow-matching policy on a binary "advantage token"** — an "again / not-again" (high-advantage / low-advantage) discrete token — and train the policy to imitate actions *conditioned on that token*. At inference, condition on the **"again" (high-advantage) token** to steer the flow policy toward high-value actions.
  - **Why this and not the obvious alternative.** The naive way to push value into a flow/diffusion policy is to backprop a Q-gradient through the sampler, which needs the policy's **flow log-likelihood** — expensive and unstable for flow matching. Advantage-conditioning **needs no flow log-likelihood**: the value's only job is to *label* each stored `(z,a)` as high- or low-advantage (a token), and the policy learns the value-conditioned action distribution by ordinary conditional flow-matching imitation. Cheap, stable, and it leaves the flow expert's training objective untouched.
  - **How the advantage token is computed (starting point to tune).** Use the BG value: `advantage(z,a) = Q(z,a) − V(z)` (with `V(z) = Q(z, π(z))` or the distributional head's expectation under the current policy). **Binarize** against a running threshold (e.g. the per-state median / a positive-advantage cutoff over the batch) into `{again, not-again}`. The **distributional (binned) value head above makes this binarization more robust** than a raw scalar twin-Q (cleaner advantage estimate, fewer threshold artifacts) — the two RECAP pieces are designed to compose.
  - **Compose, don't replace.** Keep **SimbaV2/CrossQ normalization** for the critic (high-UTD stability) and the **distributional head** above — RECAP's extraction sits *on top* of them: normalized + distributional critic → advantage token → advantage-conditioned flow policy. *(RECAP is current SOTA generation — verify the exact version/availability at build time, this moves fast.)* Threshold, token granularity (binary vs a few advantage bins), and the conditioning mechanism are all **starting points to tune**.

## Step 9 — Imagine-and-feel + the BG cache `bet`

```
# decision loop (System 2)
for each candidate action-seq A (MPPI/CEM, then learned search):
    z = z_t
    value = 0
    for k in 1..H:
        ẑ = world_model(z, A[k]);  tag ẑ = imagined        # writes to shared workspace
        value += γ^k · (amygdala(ẑ) + drives(ẑ) − pain_head(ẑ)) # feel imagined futures (pain_head = learned, Step 5 G3)
        z = ẑ
    value += γ^H · Q(z, π(z))                              # terminal value
choose A* = argmax value;  act with A*[1]                  # only the REAL post-action state drives the body
```

- **Gated on world-model faithfulness (Step 2).** This whole loop is **only enabled while the faithfulness acceptance gate passes** (open-loop `z`-error at `H` *and* feeling-consistency on rollouts — Step 2 G4). On a gate failure, **disable imagine-and-feel and fall back to reactive real-only control** until the world model recovers. The amygdala feels imagined futures only through a model it has been certified to trust.
- **Planning config — the MPPI stage (starting points to tune).** Horizon `H = 15`; discount `γ = 0.99` (matches the value-learner and warm-start in Step 8); `N = 512` sampled action-sequences per step; `3` optimization iterations (refit the sampling distribution each iter); softmax/Boltzmann **temperature** `λ` over returns when reweighting elites (**starting point** ~0.1; lower = greedier); keep the **top-k elites** (**starting point** `k = 64`, ~⅛ of `N`) to refit the mean/covariance for the next iteration. These are all starting points, not measured optima.
- The reality-tag **gates actuation** (imagined states never move the body) but **does not block gradients** through the amygdala's scoring of imagined states — that's how planning learns.
- **Gaze and voice are action dims too — scored by imagine-and-feel, not special-cased.** The candidate action sequences `A` include the **non-manipulation action dims** (where to look, whether to speak), and the world model conditions `ẑ` on them like any other action. So **"is it worth looking there?" gets answered by imagine-and-feel**: the world model rolls `z_t` forward *conditioned on a gaze action*, and the predicted-`z` for that gaze carries its expected **information gain** (e.g. drop in NE/uncertainty, Step 8 — a glance that resolves surprise scores well), so attention is planned by the same feel-the-future loop, not a bolted-on saliency rule. **Voice in v1 is the exception:** the `voice_gate` (whether/when to vocalize) is **teleop-supervised** — trained from human demonstration of when to speak — rather than discovered by imagine-and-feel, because there is no clean self-generated felt signal for "should I talk" yet. Treat gaze as fully planned, voice-gating as supervised-for-now.
- **Cache (System 1):** distill a fast policy/value head to predict the planner's choice. **Transition rule:** start with MPPI; if a decision exceeds your latency budget (e.g. >500 ms at a 10 Hz decision rate), train a **learned-search** stage on MPPI rollouts and switch once it reaches ~80 % of MPPI's value on held-out states.
  - **The value→policy link the cache distills (cross-ref Step 8, RECAP).** The System-1 policy here is the **flow-matching action expert**, and the way the BG value shapes it is **advantage-conditioned policy extraction** (Step 8, borrowed-from-PI/RECAP): the cache isn't just regressing the planner's argmax — it's a flow policy **conditioned on the advantage token**, so distilling MPPI/learned-search choices means labeling cached `(z,a)` with their advantage and training the flow policy to reproduce the high-advantage ("again"-token) action distribution. This is the concrete bridge from the imagined-and-felt value back into the fast policy — no flow log-likelihood needed.
  - **Frontier — the learned-search stage: `EfficientZero V2` primary, `Sampled MuZero` alternative.** Not generic "MuZero/TD-MPC2". The **primary learned-search default is EfficientZero V2** (the sample-efficient MuZero line — strong on limited data, applicable to continuous control, which is the regime here since the agent learns by living). **Sampled MuZero** (MuZero extended to large/continuous action spaces by searching over *sampled* actions) is the alternative for very high-DoF action spaces; TD-MPC2 remains a viable latent-planning fallback. *(These are bleeding-edge and move fast — verify the exact model, version, and availability at build time rather than asserting any version as settled.)*
  - **Smooth real-time action chunking — train-time-conditioned RTC, borrowed-from-PI. `standard`** A flow-matching policy emits actions in **chunks**, and naively stitching chunks at the latency boundary produces jerk/discontinuity while the next chunk is still being generated. **Default to Real-Time Chunking (RTC) in its *train-time-conditioned* variant** — i.e. the policy is **trained to condition on the already-committed prefix of the previous chunk** so the next chunk continues it smoothly — **over generic inference-time inpainting** (the older approach that hard-constrains/inpaints the overlap at sampling time with no training-time awareness). Train-time conditioning is the more robust default because the policy *learns* the seam rather than having it imposed at inference. This is the mechanism that lets the System-1 cache run the flow expert at control rate without stutter. *(RTC is current SOTA generation — verify the exact version/availability at build time, this moves fast.)* Chunk length and overlap/prefix size are **starting points to tune.**

**Co-training validation — Cosmos Policy (NVIDIA) validates the imagine-and-feel + BG-cache structure. `standard` (structure), distinctive bets kept.** NVIDIA's **Cosmos Policy** independently validates the **co-trained policy + world model + value as ONE model** doing **best-of-N value selection at inference** — which is exactly the *structure* of our imagine-and-feel loop (one model proposes, imagines, and scores; pick the best by value). **Adopt the co-training STRUCTURE** (policy/world/value in one co-trained model, best-of-N value selection at decision time) as external corroboration that this shape works. **But keep our distinctive substrate and objective — we do NOT take theirs:** (1) their **value is a hand-written task scalar**, whereas ours is the **felt reward** (the whole thesis — `feeling`, no external reward); (2) their planner runs **best-of-N in pixel space, which is substantially slower** than a latent planner (some reports put the gap roughly an order of magnitude — *treat the exact figure as unverified and re-check at build*), whereas ours is the **JEPA / latent-only world model** (Step 1–2) precisely to avoid that cost. So: borrow the *that-it-works-as-one-co-trained-model* structure; keep the JEPA substrate + felt objective intact. *(Cosmos Policy is current SOTA generation — verify the exact version/availability at build time, this moves fast.)*

**Honest flag:** multi-leg search is the real bottleneck — expect this to be the hardest engineering after the world model.

## Step 10 — Continuous consolidation (no sleep phase) `standard`

- One replay buffer; **continuous high-UTD updates** (ratio ~5–20) for world model, value, and caches *while acting*. **Experience replay is perpetual sleep** — no distinct phase. (Add one only if the deployed body genuinely can't learn while acting.)
- **Replay buffer spec (starting points to tune).**
  - **Capacity:** a large ring buffer — **starting point** ~1–5M transitions (size to your storage; `z_t` latents are far cheaper than raw frames, so prefer storing **latents + actions + felt-signal channels**, not pixels).
  - **Eviction:** **FIFO** by default (oldest out). If catastrophic forgetting bites the consolidated skills (below), carve out a small **protected/reservoir partition** that evicts via reservoir sampling instead of FIFO, so old skills keep a foothold in the data.
  - **Sequences vs transitions:** store **overlapping sequences** of length `L` (the world model and the learned search need temporal context — **starting point** `L = 16–64`, ≥ the planning horizon `H = 15`); the value learner samples **sub-transitions** out of those sequences. So: one sequence-keyed buffer, two readers (sequence reader for the world model, transition reader for the Q-nets).
  - **UTD split:** the global UTD (~5–20) is **divided across consumers** — **starting point** world model ~UTD/2, value ~UTD/2, BG cache as spare. Tune the split by which component is the current bottleneck (usually the world model early, the value later).
- **Optional affect-prioritized replay:** `priority = |TD_error|` (standard PER); switch to `(|TD_error| + |feeling|)/2` if strongly-felt events show flashbulb-consolidation benefit.

### Continual learning — keeping the forever-policy alive `bet` (this is the weakest area — flagged)

High-UTD updates that run *forever* are exactly the regime that degrades two ways. This is the least-proven part of the build; treat the mechanisms below as the current best known tools, not a solved problem.

- **Plasticity loss (the network slowly stops being able to learn).** High UTD + non-stationary `feeling` causes dormant/saturated units and a collapsing effective rank — the agent gets *unable to adapt* even though gradients keep flowing. Counter with **periodic/soft resets and plasticity injection:**
  - **ReDo** — periodically detect **dormant neurons** (near-zero activation over a window) and **reinitialize just those units** (and their outgoing weights), leaving learned structure intact. Run on a slow schedule (**starting point** every ~50–200k updates).
  - **Soft resets / shrink-and-perturb** — periodically pull weights a fraction toward fresh init (`w ← (1−σ)·w + σ·w_init`, small `σ`) to restore plasticity without wiping the policy. A full reset of the **last layers** (keeping the trunk/world-model features) is the heavier fallback.
- **Catastrophic forgetting (new skills overwrite consolidated ones).** Add an explicit **retention mechanism** on skills that have already consolidated. **Default stack: replay-CLEAR-style retention + periodic resets (above). EWC is dropped** — Fisher-weighted EWC is known to underperform / fail in this regime (in our own prior tests it was no better than plain finetune), so it is **not** part of the recipe.
  - **Replay-CLEAR-style retention (the workhorse)** — keep a slice of replay biased toward *old* experience (the protected reservoir partition above) and add a **behavioral-cloning / KL anchor** to the agent's own past policy on those samples, so updates can't silently drift the old behavior (this is the CLEAR recipe: behavior-cloning + value-anchoring on replayed old data).
  - **Honest flag:** in our own prior tests plain *replay* was the strongest mechanism, sleep-style consolidation underperformed it, and EWC failed outright (see the continual-learning notes); so **rely on strong replay-CLEAR-style retention plus the plasticity resets above**, and **measure forgetting directly** rather than assuming any mechanism works at scale. This remains the weakest, least-proven part of the build — flagged as such.

## Step 11 — The life: feeling-driven open-ended learning `bet` (machinery) + `wall` (principles)

Turn off the external reward. The agent acts under `feeling` alone: imagine-and-feel for choices, habituation for curiosity, continuous consolidation for memory. **No shortcut** — the true amygdala pull and the value system are *grown here, by living, and nowhere else* (the `wall`); the planning/caching/consolidation around them is the `bet` machinery.

Run it **under hard safety bounds:** physical actuation limits (velocity/torque/workspace, hardware-enforced, non-overridable by any learned policy); the felt signal, pull, values, and imagined rollouts all **logged and inspectable** — and since the loop is **latent-only** (Step 1), that means logging the **latents and the scalars derived from them** (valence, derived-cortisol, dopamine, pain, the `real|imagined` tag), not decoded media; an offline decoder may render a latent for a human when needed, but inspection does not depend on it. Autonomy widened slowly from constrained settings; **shutdown upstream of feeling** (desires never gate the off-switch).

## Step 12 — Evaluation = falsification `bet`

No reward to benchmark — test the **divergence prediction** vs a matched baseline (same backbone + a standard novelty reward), thresholds **pre-stated**:

1. **No-reward open-ended development.** Cluster the state history into modes (k-means on trajectory features, k≈10–20); count discovered modes over fixed wall-clock. Example threshold: felt agent ≥ 10 modes vs baseline ≤ 5.
   - **Trajectory-feature vector (define it concretely — starting points to tune).** Cut the state history into fixed windows (**starting point** ~2–5 s) and summarize each window into one feature vector by concatenating: **(a)** mean-pooled workspace `z_t` over the window (what it was perceiving — semantic content), **(b)** proprio/action statistics (mean + std of joint targets / end-effector pose — what the body was *doing*), and **(c)** felt-signal summary (mean dopamine, mean cortisol, mean pain over the window — the affective character). **Standardize** each block (per-channel z-score) before concatenation so no block dominates the distance metric.
   - **Operational discovered-mode estimator.** Run **k-means** (`k≈10–20`) on these standardized trajectory-feature vectors over the run; a "**discovered mode**" = a cluster that (i) holds **≥ a minimum fraction of windows** (**starting point** ≥1–2 % of total, to drop noise clusters) **and** (ii) is **first meaningfully occupied after a warm-up period** (so you count *newly explored* behavior, not the startup distribution). Count discovered modes for the felt agent vs the matched baseline over identical wall-clock. (Choose `k` by silhouette/elbow on held-out windows so the count isn't an artifact of `k`.)
2. **No wireheading.** Install an exploitable pleasure source; measure `time-on-stimulus / total`. Example threshold: felt < 20 % vs reward-maximizer near-saturating.

**Honest flag:** pre-committing thresholds *helps* but does **not** eliminate unfalsifiability. Disputes will still arise — was a divergence the amygdala or just world-model fidelity? Was the scale really enough, or a false negative? The thesis is intrinsically ambitious (embodied, large-scale, long-horizon), and that ambition *is* why the unfalsifiability caveat stands. Naming it is the honesty; pre-stated thresholds are the most you can do, not a proof.
